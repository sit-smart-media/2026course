# LLM × ASR loop の代表的パターン（4類型）

## ASR 再スコアリング loop（最も古典）
流れ

ASR が N-best 仮説を生成
LLM が「意味的に良い」候補を選択・修正
再度 ASR / LM / acoustic score で整合性確認
スコア低下なら別候補へ

📌 音響制約が強く、LLMの暴走を抑えやすい

ASR rescoring + confidence estimation 系
ELECTRA / LM を用いた誤り検出
 [ASR RESCOR...N WITH ...]


## Error Detection → Correction → Verification loop（近年主流）
3段構成（RLLM-CF）

- 誤り検出（ASR confidence / token-level uncertainty）
- LLMによる局所修正（Chain-of-Thought）
- 修正後テキストを検証: 音声と対応しない変更を reject

この構造は
「LLMが正解部分を壊す」問題を明確に回避するために設計されています。 [arxiv.org]
👉 重要点

LLMは 自由生成 ではなく
「音声に由来する誤り候補のみ触る」

## Confidence-aware control loop（制御理論的）
ASR が出す token / word-level confidence を用いて、

- confidence 高 → LLMは「触らない」
- confidence 低 → LLM介入を許可
  
という ゲート付き loop。
ASR confidence estimation は
下流制御に有効だが完全ではないことも評価されています。 [arxiv.org]

## 音響的逆検証 loop（最も厳密）
LLM → Text → “音声的にあり得るか？” を検証
手法例：

LLM修正後テキストを

- phoneme 化
- alignment / CTC loss で再評価


音響尤度が著しく下がる変更は reject

CTC 系 ASR は
「無音から単語を生成できない」構造的制約があり、
hallucination 耐性が高いことが知られています。 [nyosegawa.com]

# 典型的な loop の疑似図
Plain TextAudio 
↓
ASR (N-best + confidence) 
↓
LLM (error-focused rewrite) 
↓
Acoustic / ASR verification 
↓ 
├─ ok        → finalize 
└─ mismatch  → constrain + retry (loop)その他の行を表示する

このとき LLM には必ず：

「語を追加してはいけない」
「音素的に近い置換のみ許可」
「変更理由を説明させる」

などの 厳しい拘束条件を与えます。

# なぜこの loop は効くのか（理論的要点）

✦ LLMの弱点

真理ではなく 尤度最大化
入力を「正しい観測」と仮定

✦ ASR／音響モデルの強み

観測拘束（観測がないと出せない）
CTC / alignment による物理的制約

👉 このため、

LLMを提案器、ASRを裁判官にする

構図が最も安定します。

# 研究的まとめ（重要）

- LLMを 最後に置くほど危険
- Loopなしの LLM-ASR 補正は必ず破綻する
- Verification-in-the-loop は必須構造

この考え方は医療・議事録・法廷・教育音声など「誤りを生成してはいけない領域」では事実上の標準になりつつあります。
