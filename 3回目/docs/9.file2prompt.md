# ASR誤り対策に関しては、RAGより「専用プロンプト（ルール）をファイル化して固定運用」の方が再現性・制御性・実効性の点で優れていることが多い

- 理由1
① 問題の性質に合っている（RAG vs プロンプト）
ASR誤り訂正で欲しいのは：

✅ どこを疑うべきか
✅ 何を変えてはいけないか
✅ どういう誤りが多発するか

これは
👉 知識検索ではなく
👉 推論制約・振る舞い制御の問題です。
RAGは：

「正しい情報を外から足す」 仕組み

専用プロンプトは：

「どう考え、どう直すかを縛る」 仕組み

ASR後処理に必要なのは後者。

- 理由2. RAGは「入力誤り」を前提にできない
RAGは基本的に、
誤ったASR文
→ 検索
→ LLMが整合的に書き直す

ですが、LLMは誤り箇所を推定できないため：

勝手に書き換える
本来正しい単語まで修正する
誤りを意味的に正当化する

になります。
一方、専用プロンプトでは：

「音声由来なので幻覚禁止」
「確信がなければ変更しない」
「ありがちな誤りカテゴリのみ修正対象」

とルール化できる。


## 実際に効く「ASR後処理プロンプト」の設計観点
① 役割固定（最重要）
あなたはASR後処理専門の校正エンジンです。
文章を「改善」してはいけません。
目的は、音声認識誤りの可能性が高い箇所のみを最小限修正することです。

👉 これだけで「作文化」をかなり抑止できます。

② 変更禁止ルールを先に書く
以下は絶対に変更してはいけません：
- 文の構造
- 話者の口調
- 意味が一義に確定しない語

LLMは「何をしてはいけないか」を書かないと必ずやり過ぎます。

③ 修正対象を限定
修正を検討してよいのは次のケースのみ：
- 明確な同音異義誤り（例：抗体↔交代）
- 文脈的に破綻している専門用語
- 数値＋単位の不整合
👉 自由度を削る = 精度が上がる

④ 「判断不能時の振る舞い」を明示

音声情報がないため断定できない場合：
- 修正せず原文を維持
- コメントとして候補を示す（本文は変更しない）

これにより 誤修正率（false positive）が激減します

## ファイル化するメリット（RAGに対する優位点）
✅ 再現性

同じ入力 → 同じ出力
embeddingの揺れなし
検索失敗なし

✅ デバッグ可能

「このルールが効いた / 効かなかった」が追える
ASR改善サイクルに乗せやすい

✅ モデル更新耐性

RAGはモデル側挙動変化に弱い
ルールプロンプトは比較的安定

👉 研究用途・プロダクション両方に向く

## RAGを完全に捨てるべきか？
完全に捨てる必要はありませんが、用途はこれだけに限定されます：

✅ 固有名詞・製品名・専門語リストの参照
✅ 候補提示（本文は変えない）

それ以上やると ASR訂正ではなく“再解釈” に変質します。

まとめると、
- ASR誤り訂正 ≠ 知識検索
- 専用プロンプトは「後段言語モデル」の代替
- RAGは補助、メインはルール＆制約
