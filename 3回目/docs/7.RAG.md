# introduction

医用音声の hallucination‑safe ASR/LLM loop では
✅ RAGは「そのまま使う」は危険
✅ ただし「役割を限定すれば使える」
❌ 決して LLMの生成用知識源として使ってはいけない

以下で なぜ危険か／どうなら使えるか を医用前提で整理します。

# そもそも RAG は何をする技術か（再確認）
RAG（Retrieval-Augmented Generation）は本質的に：

「確率的に尤もらしい生成」を
「外部テキストで強化する」技術

です。
これは：

QA
要約
説明生成

には非常に有効ですが、
❗ 「観測証拠（音声）」が最優先の医用 ASR とは目的が逆
になります。

# 医用音声で “危険な RAG の使い方”
## NG①：診断・症状・薬剤知識を RAG で補完

Audio ASR → LLM + 医療知識ベース → 医用テキストその他の行を表示する

なぜダメか

音声に無い情報でも
RAGが「根拠付きでもっともらしく」出してしまう

📌 医療ではこれは hallucination ではなく fabrication（捏造）
👉 最悪ケース：診断の自動補完

## NG②：「辞書＋RAG」で LLM を賢くする

RAG で

ICD‑10
UMLS
RxNorm
を与える

→ LLMが 音声に無い医療語を積極的に挿入
これは 辞書の誤用 + RAGの誤用 の合わせ技です。

#  医用で「使える RAG」の正しい定義
医用 hallucination‑safe 設計では、

RAGは「生成を助けるため」に使わない
RAGは「検証・分類・フラグ付け」にだけ使う

これが唯一の正解です。

# 正しい RAG の使い方（安全な3用途）
## ① ✅ 後段検証 RAG（Verification‑only RAG）
RAGを LLMの入力に渡さない。
Plain TextASR / WhisperX / CTC → text ↓RAG (医療DB) ↓「この語は医療語か？」「ハイリスクか？」「医師確認が必要か？」その他の行を表示する
✅ LLMは一切見ない
✅ 生成には影響しない
👉 分類器としてのRAG

## ② ✅ エンティティ危険度フラグ RAG
ASR結果に含まれる語を：

薬剤名？
診断名？
手技？
数値依存語？

としてタグ付けするだけ。
"text contains: METFORMIN" → tag: DRUG_NAME (high‑risk)その他の行を表示する

→ このタグがある場合のみ：

LLM修正を禁止
CTC＋人手確認必須


## ③ ✅ Human review 補助 RAG（完全分離）
医師・書記が見る UI 側で：

関連ガイドライン
医薬品説明
ICD 定義

を引くのは OK。
✅ 人間が評価
❌ LLMは触らない
👉 RAGは 意思決定支援、生成ではない。

# どうしてもダメな RAG の役割
以下は 一発アウト。

|用途|理由|
|---|---|
|診断補完|音声外情報|
|症状推定|捏造|
|医療要約|意味改変|
|表現の医療化|hallucination増幅|

# WhisperX + CTC + 辞書 + RAG の最終安全アーキテクチャ

Audio
 ↓
WhisperX (存在・時刻証明)
 ↓
LLM (STRICT, RAG無)
 ↓
CTC-ASR (音響可能性)
 ↓
Medical Dictionary (語彙制約)
 ↓
RAG (検証・タグ付けのみ)
 ↓
Accept / Flag / Human Review

RAGは一番最後、しかも生成に関与しない

一言で言うと（重要）

医用音声における RAG は「知識注入」ではなく、「危険検出センサー」
